# Team Ground Rule (이루리) v0.9

## 커뮤니케이션
1. 연락은 6시간 이내 응답 (단. 야간 24:00 ~ 05:00 예외)
2. 해결되지 않는 문제(코드 오류 등)는 혼자 끌지 말고 공유
3. 불만 사항은 바로바로 이야기하기
4. 제출 후 수정이 가능한 과제의 경우에는 별도의 리더 컨펌 없이 각자 제출
5. 중요한 공지/이슈는 팀 톡방에 공유
   

## 일정
- 부득이하게 마감 기한을 미루게 될 경우, 최대로 미룰 수 있는 시간은 48시간
- 마감 기한을 미룰 시, 마감 기한 하루 전까지 톡방에 사유와 함께 공지
- 연속으로 최대 2번까지 미룰 수 있음. 이후에는 아래의 벌금 규정에 따라 벌금이 부과됨
- 벌금: 2시간 지연당 3,000원
- 단, 이해될 수 있는 사유에 관해서는 Ground Rule을 적용하지 않음. 
- 이해될 수 있는 사유: 갑작스러운 질병/응급, 가족 관련 긴급 상황, 시험

## 협업 규칙
1. main 브랜치는 항상 깨끗하게 유지한다.
   - main은 시연 가능한 코드만 들어간다.
   - main에서 바로 작업하지 않는다. (금지)
2. 각자 맡은 기능은 반드시 "브랜치 따서" 개발한다.
   - 브랜치 이름 규칙:
     * feature/push-parsing
     * feature/crawling
     * feature/policy-matching 
3. 모든 개발은 로컬(VS Code)에서만 한다
   - GitHub 웹에서 직접 코드 수정 X
   - 로컬에서 코드 실행 -> 확인 -> commit -> push하는 방식으로 진행
4. main에 올릴 때는 반드시 Pull Request(PR)를 사용한다.
   - 작업 완료 -> GitHub에서 PR 생성
   - 코드 리뷰 후 merge
   - 본인 브랜치를 본인이 merge해도 OK
5. 팀 공통 개발환경(venv+requirements.txt)반드시 사용한다.
   - Python 버전: 3.10.x 통일
   - 카독/지라에 별도 첨부한 개발환경 문서 확인
6. 폴더 구조는 절대 임의로 바꾸지 않는다. 필요한 파일만 각자 폴더 안에서 추가한다.
7. 기능 테스트 후 push한다. (깨진 코드 금지)
   - Streamlit 데모 실행 확인
   - 파싱/크롤링 최소 1회 테스트 후 push
   - 오류 난 상태로 push 절대 금지
8. 커밋 메시지는 짧게 기능 중심으로 쓴다
   예:
   - "Add push parsing regex"
   - "Implement policy crawler for Seoul"
   - Update matcher logic"
   "수정함", "최종", "진짜최종" 등의 표현은 쓰지 않는다.
9. 작업 시작 전 반드시 pull -> 최신 main 반영
10. 작업 단위는 작게, PR은 자주
    - 200줄짜리 대형 PR 금지
    - 작은 단위로 기능 나눠서 merge
